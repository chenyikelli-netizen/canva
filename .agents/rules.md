# Canva Brand Sentinel — 專案規範

## 專案定位

本專案為 **Canva 品牌輿情監控與分析系統**，服務對象為股東，目的是：

- 確認投資效益
- 掌握品牌發展脈絡與趨勢走向

系統以全自動排程運行，每日產出結構化摘要報告，並透過通訊軟體推送。

---

## 監控目標

| 項目     | 內容                                    |
| -------- | --------------------------------------- |
| 追蹤品牌 | **Canva**（含相關產品線與子品牌） |
| 關鍵字   | Canva、canva.com、Canva AI、Canva 設計  |

> [!IMPORTANT]
> 關鍵字清單需持續維護。當 Canva 推出新產品或功能時，應即時更新。

---

## 資料來源

依優先順序排列：

1. **Tavily 首選整合搜尋** — 即時抓取新聞、科技網誌、論壇的綜合分析
2. **Threads** — 社群討論與綜合輿論
3. **Dcard** — 台灣年輕族群社群論壇
4. **Google 評論** — 使用者直接體驗回饋

### 資料蒐集原則

- 僅蒐集**公開資訊**，不存取任何需登入或私人的內容
- 每次蒐集需記錄：來源平台、原文連結、發布時間、作者（公開帳號名稱）
- 資料保留期限：至少 90 天，供趨勢比對使用

---

## 分析框架

### 主題分類體系

所有蒐集到的資訊須歸入以下主題類別：

| 主題類別   | 說明                                     | 範例                         |
| ---------- | ---------------------------------------- | ---------------------------- |
| 產品功能   | 對 Canva 產品功能的討論、評價、建議      | 新功能上線反應、功能缺陷回報 |
| 定價與方案 | 訂閱價格、免費/付費方案比較              | 漲價討論、CP 值評估          |
| 使用者體驗 | 操作介面、易用性、學習曲線               | 新手教學、使用心得           |
| 競品比較   | 與 Figma、Adobe、PowerPoint 等工具的比較 | 工具選擇討論                 |
| 品牌形象   | 品牌認知、市場定位相關討論               | 品牌信任度、企業形象         |
| AI 相關    | Canva AI 功能與 AI 策略相關話題          | Magic Design、AI 生圖        |
| 教育與企業 | 教育版、企業版（Canva for Teams）應用    | 學校導入、團隊協作           |
| 市場動態   | 併購、上市、融資、營收等商業資訊         | IPO 傳聞、估值討論           |

### 情緒標籤

每筆資料須標記情緒傾向：

- `正面` — 推薦、讚賞、滿意
- `中性` — 客觀描述、提問、資訊分享
- `負面` — 抱怨、批評、不滿

### 品牌發展趨勢指標

報告需追蹤以下趨勢面向：

1. **聲量趨勢** — 各平台提及次數的變化
2. **情緒變化** — 正/負/中性比例的走勢
3. **熱門主題** — 當期最多人討論的話題
4. **風險訊號** — 負面聲量異常升高或特定議題發酵

---

## 報告規格

### 頻率

每日一次，建議於每日 **08:00 (UTC+8)** 前完成推送。

### 報告結構

```
📊 Canva 品牌日報 — {日期}

■ 今日摘要
  - 總蒐集筆數 / 各平台分布
  - 整體情緒分布（正/中/負 比例）

■ 熱門主題 TOP 3
  - 主題名稱 + 簡要說明 + 代表性原文摘錄

■ 關鍵洞察
  - 與投資效益相關的觀察
  - 品牌發展趨勢解讀

■ ⚠️ 風險警示（如有）
  - 異常負面輿論
  - 需關注的議題

■ 週趨勢（每週一附加）
  - 本週 vs 上週聲量對比
  - 情緒趨勢圖表
```

### 輸出格式

- 報告主體為 **Markdown 格式**，便於通訊軟體渲染
- 趨勢圖表以文字圖表或圖片形式呈現

## 技術準則

### 架構原則

- 模組化設計：蒐集、分析、報告、通知各為獨立模組
- 每個模組應可獨立測試與替換
- 設定檔與程式碼分離，敏感資訊（API Key 等）使用環境變數

### 程式碼規範

- 使用 **繁體中文** 撰寫註解與文件
- 變數名與函式名使用 **英文**，採 snake_case 命名
- 所有對外請求需設定 timeout 與 retry 機制
- 錯誤處理須記錄完整 log，包含時間戳與錯誤上下文

### 資料處理

- 蒐集的原始資料與分析結果分開儲存
- 使用 JSON 或 SQLite 作為資料儲存格式
- 資料須包含 `collected_at` 時間戳

---

## 開發階段

```
Phase 1: 資料蒐集模組（Search Tools 串接）
Phase 2: LLM 分析模組（主題分類 + 情緒分析）
Phase 3: 報告生成模組（日報 + 週報）
Phase 4: 通知推送模組（LINE / Telegram Bot）
Phase 5: 排程與自動化（定時觸發全流程）
```

> [!NOTE]
> 每個 Phase 完成後應可獨立驗證，確保該模組功能正確再進入下一階段。
